## Named bugs referenced by PLAN.md Phase 1

- **MC infinite loop**: `MultipleChoice::generateOptions` (`StudyingMethods/MultipleChoice.cpp:96-102`)
  — for sets >4 cards, the distractor loop demands 3 distinct wrong Values; Value has no
  UNIQUE constraint, so duplicate answers make it spin forever and hang the UI.
- **First-card skip**: `Interface/FlashcardPage.cpp:16,141-147` — `showNextFlashcard()`
  calls `goToNextQuestion()` before displaying, so index 0 is never shown.
- **Finish button never shown**: `Interface/FlashcardPage.cpp:25,31,136` — hidden in three
  places, no `show()` anywhere; flashcard sessions cannot end.
- **Shuffle-before-pad**: `MultipleChoice.cpp:108-112` (and Inverse twin) — small sets pad
  with "" AFTER `std::shuffle`, so blanks always occupy the last option slots and the
  correct answer is confined to the first positions.
- **Apostrophe breakage**: `User/UserSession.cpp:151` etc. — SQL built by string
  concatenation of user text; a card containing `'` breaks the INSERT.
- **Edit destroys history**: editing a card deletes and re-inserts it, zeroing
  `TotalCorrect`/`TimesAsked`.
- **MainWindow leak** (fix worth porting from origin/shortcuts): `Interface/MainWindow.cpp:120-125`
  on main allocates a parentless `QListWidget` per created set into a never-read map; the
  shortcuts branch deletes this and adds the app's only menu bar.

## Per-area detail (agent findings, verbatim)


### database

**Summary:** DatabaseManager is honest student work, not fake: a ~140-line thin sqlite3 wrapper with a correctly executed (if naive and thread-unsafe) singleton, real GoogleTest coverage running in genuine Ubuntu+macOS CI with clang-format enforcement, and a deliberate TESTING/resetInstance testability hook that the tests actually use. The headline resume claims mostly check out - SQLite backend, per-question TotalCorrect/TimesAsked tracking, 36 tests, C++17/Qt6 are all real - with one outright fabrication: there is no user authentication or session management anywhere in the codebase, just a misleadingly named UserSession singleton over one local DB file. The engineering quality is mediocre where it counts: no parameter binding exists in the entire data layer so all SQL is built by string concatenation (injectable via user-entered set names and flashcard keys), zero transactions, an error-conflating isTableEmpty, a leaked handle on failed open, and a debug printDatabaseTable(\"set_names\") leftover shipped inside isTableEmpty on main. Verdict: real but junior - the infrastructure (tests, CI, lint) is the strongest part, the database code itself would not pass a professional review."

**Bugs:**
- Database/DatabaseManager.cpp:45-48 - executeQuery has no null-db guard (unlike executeQueryWithResults at cpp:70-73). Calling it before openDatabase() passes nullptr to sqlite3_exec, which is API misuse and can crash. Inconsistent defensive posture within the same file.
- Database/DatabaseManager.cpp:29-33 - on sqlite3_open failure the handle is never closed. SQLite docs require sqlite3_close() even when open fails; this leaks the handle AND leaves the private db member non-null pointing at a dead connection, so subsequent calls think the DB is open.
- Database/DatabaseManager.cpp:18-25 - getDatabaseManager(databaseName) silently ignores databaseName on every call after the first. Requesting a different database returns the connection bound to the original name with no error. Tests only survive this via the TESTING-only resetInstance; production code has no escape hatch. Instance is also never deleted outside TESTING (leak by design) and lazy init is not thread-safe.
- Database/DatabaseManager.cpp:106-129 - isTableEmpty returns false both when the table has rows AND when the query fails (cpp:116, :125), conflating 'error' with 'not empty'. The header doc (h:74-78) does not document the error behavior.
- Database/DatabaseManager.cpp:97 and :108 - SQL injection / breakage vector: tableName is concatenated raw and unquoted into 'SELECT * FROM ' and 'SELECT COUNT(*) FROM '. Table names upstream are user-entered study-set names (UserSession.cpp:99 creates a table per set name), so a set name with a space, quote, or SQL token breaks or injects. Identifiers cannot be bound as parameters, but they are neither quoted nor validated here (UserSession quotes with \" in its own queries; DatabaseManager's own methods do not).
- Database/DatabaseManager.cpp:50 and :78 - std::cerr << errMsg where errMsg can be nullptr (sqlite3_exec leaves *errmsg NULL in some failure modes, e.g. OOM); streaming a null char* is undefined behavior. Low likelihood, real UB.
- Enabled by this class's string-only API, confirmed injection upstream: UserSession.cpp:202 builds WHERE Key = '<key>' by concatenation, so any flashcard key containing an apostrophe (e.g. "don't") breaks the accuracy UPDATE - the very feature claim 1 rests on.

**Slop:**
- Database/DatabaseManager.cpp:121 - hardcoded printDatabaseTable("set_names") inside isTableEmpty(): a debug leftover shipped to main. Every emptiness check on ANY table dumps the entire app-specific set_names table to stdout, and in test/other databases (e.g. the :memory: DB in tests/testDatabase.cpp:69-75) it spews 'SQL error: no such table: set_names' to stderr. Also a layering violation: the generic DB wrapper hardcodes an application table name.
- Database/DatabaseManager.h:52-53 - doc comment claims '@return True if the query was successfully executed, false otherwise' on executeQuery, which returns int (a raw sqlite rc, compared against SQLITE_OK by callers). Copy-paste doc rot; the @brief at h:49 and h:57 are also identical word-for-word for two different functions.
- Database/DatabaseManager.cpp:95-104 - printDatabaseTable: a std::cout console-print method in the data layer of a Qt GUI desktop app. Its only production call site is the debug leftover at cpp:121. Dead weight masquerading as API surface.
- Database/DatabaseManager.cpp:108-112 - the only sqlite3_prepare_v2 in the file never binds a single parameter; the table identifier is still string-concatenated into the query. It has the appearance of prepared-statement hygiene with none of the substance. The class offers no parameter-binding API at all, forcing every caller (see UserSession.cpp:151, :202) into string-concatenated SQL.
- Class naming slop adjacent to my files: 'UserSession' (friend at DatabaseManager.h:104) implies sessions/users; it is a singleton table-manager for one local SQLite file. This naming is what props up the false 'user authentication and session management' resume claim.

**Worth keeping:**
- The TESTING-gated resetInstance design (DatabaseManager.h:97-102, DatabaseManager.cpp:132-140) wired to target_compile_definitions(TestExecutables PRIVATE TESTING) in CMakeLists.txt:101, with friend test fixtures (h:105-106) actually used by tests (tests/testDatabase.cpp:8-20). This is real, deliberate testability engineering, not decoration.
- Deleted copy constructor and copy assignment (DatabaseManager.h:94-95) plus private explicit constructor (h:92) - correct, textbook singleton hygiene.
- Idempotent closeDatabase (DatabaseManager.cpp:37-43, nulls db after close) and a destructor that closes the connection (cpp:13-16); double-close is even explicitly tested (tests/testDatabase.cpp:37-41).
- Correct sqlite C-API error hygiene in the happy paths: sqlite3_errmsg for diagnostics (cpp:31, :114, :124), sqlite3_free on error messages (cpp:51, :79), sqlite3_finalize on the statement (cpp:128).
- executeQueryWithResults (cpp:67-82): null-db guard plus a simple generic vector-of-maps result shape via the callbackStore callback (cpp:56-65). Modest but genuinely useful abstraction, and it is covered by a real test (tests/testDatabase.cpp:54-65).

### study-methods

**Summary:** The StudyingMethods area is real, working student code, not vapor — there is a genuine accuracy-biased question-selection mechanism wired to per-question SQLite stats, competent modern-C++ randomization, deliberate small-set handling, and unit tests. But the headline claim is inflated: 'adaptive study sessions' is a one-shot bottom-N-by-lifetime-accuracy plus random sampling at quiz start, with zero spaced repetition (no intervals, ease factors, due dates, or within-session re-queuing), and the flashcards mode is purely sequential and never touches accuracy at all. The most damning quality signal is that InverseMultipleChoice is a 122-line verbatim copy-paste of MultipleChoice with .first/.second swapped on ~8 lines, and that copy-paste both preserved a dead padding loop and introduced an asymmetric real bug: MultipleChoice::generateOptions can infinite-loop and hang the app on any 5+ card set with duplicated answer values, while the inverse variant escapes only because Keys happen to be UNIQUE. Verdict: roughly 60% honest engineering with sensible heuristics and tests, 40% duplication and overstated 'adaptive' marketing."

**Bugs:**
- MultipleChoice.cpp:96-102: infinite loop / UI hang. When setSize > 4, the while loop demands 3 distinct distractor Values different from the correct answer, but the Value column has no UNIQUE constraint (UserSession.cpp:99: 'Key TEXT UNIQUE, Value TEXT'). A 5+ card set where answers repeat (e.g. four cards answered 'mitochondria' and one 'ribosome') has fewer than 3 distinct wrong values for some questions, so the loop never terminates and the app freezes. Duplicate answers are common in real flashcard sets. InverseMultipleChoice.cpp:96-102 is the same code but samples Keys, which ARE unique, so it is safe only by accident.
- Flashcards.cpp:51: setToLastIndex does 'currentIndex = keyValues.size() - 1' with no empty-set guard; for an empty set size_t 0-1 wraps and truncates into uint32_t 4294967295. getQuestion then returns "" (guarded), but goToPrevQuestion (Flashcards.cpp:37) will happily 'succeed' ~4 billion times from that state.
- MultipleChoice.cpp:99 and :104 (same in InverseMultipleChoice.cpp): generateOptions indexes keyValues[currentIndex] in the setSize>4 branch with no bounds check. Constructing MultipleChoice(set, 0, 0) on a >4-card set yields empty keyValues and undefined behavior on the first generateOptions call. The current UI happens to always pass totalPairs/2 (MCPage.cpp:17-20) so it is latent, but the public API permits it and nothing tests it.
- MultipleChoice.cpp:108-112 (and Inverse): padding with "" happens AFTER std::shuffle, so for sets smaller than 4 the blank options always occupy the last tuple slots — the correct answer is confined to the first positions (in a 2-card set, always slot 1 or 2), which biases guessing.
- MultipleChoice.cpp:8-14 / InverseMultipleChoice.cpp:8-14: pair_hash uses plain XOR of the two hashes — symmetric, so (a,b) and (b,a) collide and (x,x) hashes to 0. Harmless at flashcard scale but a textbook weak-hash mistake, duplicated in two files.

**Slop:**
- MultipleChoice.cpp vs InverseMultipleChoice.cpp: 122-line file copy-pasted wholesale; diff shows the only substantive differences are ~8 lines swapping .first/.second (e.g. MultipleChoice.cpp:50 vs InverseMultipleChoice.cpp:50, :57, :65, :99, :104). The headers are likewise 95% identical (MultipleChoice.h vs InverseMultipleChoice.h, 33 diff lines all renames). One class with an orientation flag, or a shared base implementing the algorithm, was the obvious design. The duplicated pair_hash struct (MultipleChoice.cpp:8-14, InverseMultipleChoice.cpp:8-14) is part of the same copy-paste.
- Flashcards.h:30-36: Doxygen comment for getQuestion says it overrides getQuestion 'to display the answer to the current question' — a copy-paste of the getAnswer comment. Documentation that is wrong, not just verbose.
- MultipleChoice.cpp:88-90 and InverseMultipleChoice.cpp:88-90: dead/ineffective padding loop — repeatedly emplacing "" into an unordered_set<string> can only ever insert one element, so the pad-to-4 intent fails for 1-2 card sets; the code only works because a second padding loop at lines 110-112 rescues it. Reveals the first loop was never reasoned about or tested for its stated purpose.
- StudyMethods.h:8-11: includes <iostream>, <string>, <utility>, <vector> in an interface that uses only std::string; three of four includes are unused filler.
- The README/resume word 'adaptive' itself: the code implements bottom-N-by-lifetime-accuracy + random sampling at session start (MultipleChoice.cpp:27-43) and nothing else; calling that 'adaptive study sessions to optimize learning efficiency' is resume inflation relative to what 'adaptive' means in the SRS literature (SM-2/FSRS/Leitner scheduling, intervals, ease factors — none present).

**Worth keeping:**
- The lowest-accuracy + random mix with dedup and shuffle (MultipleChoice.cpp:25-43) is a real, sensible heuristic — half remediation, half coverage — and it is genuinely wired to per-question stats recorded by updateScoresInTable. Underdescribed code, oversold claim.
- Randomization is done properly: std::mt19937 seeded from std::random_device, std::shuffle, std::uniform_int_distribution (MultipleChoice.cpp:43, 82-83, 96-97) — not rand()%n. This is above the bar for student code.
- Small-set (<4 cards) handling was deliberately designed: the setSize<=4 branch forces all cards in (MultipleChoice.cpp:27-30), generateOptions pads with empty strings (lines 85-92, 110-114), the UI hides empty options (MCPage.cpp:189-195), and there are unit tests for it (tests/testMultipleChoice.cpp:27 InitializationWithFewEntries, :56 GenerateOptionsForSmallSet).
- Study logic is plain C++ decoupled from Qt, behind a small abstract base (StudyMethods.h:17-43) with guard-checked accessors that return "" or false at boundaries (Flashcards.cpp:10-42) — simple and correct.
- All three method classes have dedicated GoogleTest files including out-of-bounds tests (tests/testInverseMultipleChoice.cpp:103,112), with friend test hooks (MultipleChoice.h:87).

### user-tests

**Summary:** The data layer is real but the resume framing inflates it: per-question accuracy tracking in SQLite genuinely works (UserSession.cpp:99,189-215,286-291), and adaptive selection exists in a basic half-worst/half-random form for the MC modes — but "user authentication and session management" is flatly false, with zero auth code anywhere and UserSession being a misnamed single-user CRUD facade carrying dead "session" fields (UserSession.h:151,156). The "35+ GoogleTest cases on Ubuntu and macOS" claim is numerically true (36 cases, real CI matrix), but roughly half are thin DB-plumbing round-trips against a shared persistent StudySets.db with cleanup hidden inside assertion tests, making the suite order-dependent and unable to survive --gtest_shuffle or filtered reruns; two "OutOfBounds" tests test nothing and assert the opposite of the documented behavior, one test has no assertions, and the entire Qt Interface/ layer (10 files) is untested and excluded from the clang-format lint. The pervasive SQL string concatenation means flashcards containing an apostrophe simply break, which is a functional bug in a flashcard app, not just a security footnote. Verdict: a legitimate student project with some real engineering (CI matrix, -Werror, the accuracy schema, the one well-isolated test suite) wrapped in two clearly padded claims (auth: fabricated; test rigor: overstated). Note: the committed README contains the audited claims; an uncommitted working-tree rewrite already walks most of them back to accurate descriptions."

**Bugs:**
- UserSession.cpp:151 (also :71, :125, :167, :198-202, :226, :247, :312) — SQL built by string concatenation of user-controlled flashcard text. Practical consequence beyond injection: any question or answer containing an apostrophe (e.g. "What's DNA?") fails to INSERT/DELETE/UPDATE in a flashcard app whose whole job is storing arbitrary text. addToStudySet returns false and the card is silently unusable.
- UserSession.cpp:55 — getStudySetSize interpolates the table name UNQUOTED (SELECT COUNT(*) FROM My Set) while createStudySet quotes it (UserSession.cpp:99). A set named with a space or hyphen can be created successfully but then getStudySetSize returns 0, so MultipleChoice (MultipleChoice.cpp:23-30) treats it as a tiny/empty set.
- UserSession.cpp:9 + tests — all UserSession/Flashcards/MC/IMC tests run against the persistent on-disk StudySets.db hardcoded in the UserSession constructor; no fixture creates or removes a temp DB. State leaks across tests and across runs: rerunning a filtered test fails (e.g. ./TestExecutables --gtest_filter=UserSessionTest.CreateAndCheckStudySet twice -> second run fails testUser.cpp:49 because 'NewSet' persists and the UNIQUE insert at UserSession.cpp:92-98 fails). The suite only passes in full-run order because cleanup is buried inside assertion tests.
- tests/testMultipleChoice.cpp:122 and tests/testInverseMultipleChoice.cpp:120 — deleteStudySet("TestSet") hidden inside the LAST test of each suite is load-bearing cleanup: testInverseMultipleChoice.cpp:99's expectation TimesAsked==1 only holds because the MC suite's final test deleted the shared 'TestSet'. Run IMC without MC's last test (shuffle/filter) and UpdateScoresInTableCorrect fails. An interrupted run poisons the next local run.
- tests/testFlashcards.cpp:4-14 — FlashcardsTest has no TearDown: the 'Math' set is permanently leaked into StudySets.db, and the fixture never resets the UserSession singleton, silently depending on the preceding suite's teardown. Its assertions (testFlashcards.cpp:19) also depend on SELECT-without-ORDER-BY returning rowid order (UserSession.cpp:226).
- Database/DatabaseManager.cpp:27-35 — openDatabase does not guard an already-open handle; it overwrites db, leaking the previous sqlite3 connection. testDatabase.cpp:33-36 (OpenDatabase) triggers exactly this: SetUp already opened the DB, the test opens it again. For :memory: the second open is also a brand-new empty database, silently discarding state.
- UserSession.cpp:189-215 — updateScore returns success for keys that do not exist (UPDATE matching zero rows returns SQLITE_OK), so a caller can 'record' answers for deleted cards with no error; untested edge.
- UserSession architecture: every method opens and closes the SQLite connection (e.g. UserSession.cpp:70-76, 150-156), which makes UserSession impossible to test against :memory: (data would vanish on each close) — the direct cause of the on-disk shared-state test design, plus double opens per call (existsStudySet guard then the real query, e.g. UserSession.cpp:191+197).
